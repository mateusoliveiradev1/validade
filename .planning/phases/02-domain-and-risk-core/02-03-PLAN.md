<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| physical observation -> domain input | Store collaborators may report concrete but imperfect observations. |
| stale confirmation -> safety state | Old observations can make a risky lot appear safe if freshness is ignored. |
| conditional resolution -> future workflow | `not_found` and `probably_sold_out` can be misused as permanent closure if not marked. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Spoofing | Physical confirmation input | mitigate | Accept only concrete status enum values and keep actor/audit persistence out of this pure phase. |
| T-02-08 | Tampering | Presence freshness | mitigate | Tests pin stale and missing confirmation as `uncertain` with `check_presence`. |
| T-02-09 | Repudiation | Conditional resolutions | mitigate | Return structured conditional markers/reasons for `not_found` and `probably_sold_out` so later audit/task phases can track them. |
</threat_model>
