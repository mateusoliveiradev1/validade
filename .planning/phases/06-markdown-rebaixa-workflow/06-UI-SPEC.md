## Visual And Interaction Contract

### Primary Surfaces

| Surface | Focal point | Required hierarchy |
|---------|-------------|--------------------|
| Hoje task list | Current markdown stage task, especially delayed application or final shelf confirmation. | Section title -> task action -> product/lote -> location/responsible/due -> alert/escalation status. |
| Markdown task panel | The one stage that can be completed now. | Stage title -> lot summary -> required decision/evidence -> blocking notice if any -> primary action -> back/review. |
| Lot detail | Physical presence state before markdown entry. | Product/lote -> current location/status -> presence notice -> markdown entry if eligible -> observe/back actions. |
| Leadership decision | Request reason and consequence of approving or rejecting. | Request summary -> approve/reject choice -> rejection reason if rejected -> confirmation. |

### Stage Model

Render exactly one active stage task at a time:

| Stage | Main label | Owner copy | Completion behavior |
|-------|------------|------------|---------------------|
| requested | Aprovar rebaixa | Lideranca local | Approve creates `Aplicar rebaixa`; reject requires reason and closes the markdown workflow. |
| approved | Aplicar rebaixa | Equipe do turno | Requires label/price evidence or explicit no-photo reason, then creates final shelf confirmation. |
| applied | Conferir etiqueta na area de venda | Equipe do turno | Requires shelf/product/location evidence or explicit no-photo reason, then marks markdown shelf-confirmed. |
| rejected | Rebaixa reprovada | Lideranca local | Terminal state; no follow-up markdown task remains visible. |
| shelf_confirmed | Rebaixa conferida na area de venda | Equipe do turno | Terminal state; show as history, not active work. |

Presence gate: if current physical presence is missing, stale, not found, probably sold out, or otherwise uncertain, the markdown entry must show `Conferir presenca antes da rebaixa` and route to the observation flow. Do not allow markdown request, approval, application, or shelf confirmation to bypass this gate.

Delayed stages:

- Delayed approval remains visible but uses a calmer warning tone than application/final confirmation.
- Delayed application and delayed shelf confirmation use critical structure: explicit `Atrasada`, stage label, responsible label, and alert/escalation status.
- Push acknowledgement never changes the stage CTA or hides the task.

Evidence interaction:

- Application evidence asks for `Registrar foto da etiqueta/preco` or an explicit no-photo reason.
- Final shelf confirmation asks for `Registrar foto da etiqueta na area de venda` or an explicit no-photo reason.
- Evidence remains structured metadata only: `photo_recorded_placeholder` or `no_photo_reason`.
- Do not capture, store, display, fixture, or reference real image URI/base64/object keys in this phase.

## Spacing Scale

Declared layout values for new Phase 6 UI must use this scale:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline helper gaps, notice text stacks |
| sm | 8px | Selection row groups, compact metadata stacks |
| md | 16px | Screen padding, card padding, default vertical rhythm |
| lg | 24px | Section gaps, stage group separation |
| xl | 32px | Bottom padding and major screen breaks |
| 2xl | 48px | Touch target minimum and large action separation |
| 3xl | 64px | Reserved for rare page-level spacing; avoid in normal mobile flows |

Exceptions: existing component internals may keep their current `captureSpacing.medium` 12px padding/gap where already used. Do not introduce new Phase 6 layout tokens outside the declared scale.

Touch targets: all Pressable actions, selection rows, date/actions, approval decisions, and evidence choices must be at least 48dp high.

Radii: use `captureRadii.small` 6px for buttons/inputs and `captureRadii.medium` 8px for notices/task rows. Do not use nested cards.

## Typography

Use four sizes and two weights for new Phase 6 UI.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label/meta | 14px | 400 or 600 | 20px |
| Body/control | 16px | 400 or 600 | 24px |
| Heading/panel title | 20px | 600 | 25px |
| Display/screen title | 28px | 600 | 34px |

Rules:

- Use 600 only for action labels, selected rows, headings, and state labels that must be scanned quickly.
- Keep body text at 16px and line-height 24px for corridor readability.
- Avoid display fonts, fluid type, decorative letter spacing, all-caps section markers, and long headings that wrap poorly on narrow phones.
- Use ASCII-normalized PT-BR copy in source files unless a touched file already uses accented copy consistently.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F7F9F6` | Screen background and calm operational canvas |
| Secondary (30%) | `#FFFFFF` | Task rows, form fields, primary content surfaces |
| Accent (10%) | `#166534` | Primary completion CTA, selected row border/background pairing, success notice text, active task highlight |
| Destructive | `#A1271D` | Rejection errors, expired/critical blocking states, destructive confirmation only |

Supporting semantic colors from `capture-theme.ts`:

| Role | Value | Usage |
|------|-------|-------|
| Muted surface | `#EDF3EE` | Summary blocks, neutral notices, inactive metadata containers |
| Pressed surface | `#E2EBE4` | Press feedback on secondary surfaces |
| Border | `#BAC8BE` | Default row, input, and notice border |
| Ink | `#132018` | Primary text |
| Muted ink | `#3E5547` | Secondary text and metadata |
| Critical surface | `#FFF0EE` | Critical or delayed application/final confirmation rows |
| Critical border | `#E8B5B0` | Critical row border |
| Warning surface | `#FFF7E2` | Approval delays, alert setup warnings, non-critical blockers |
| Warning ink | `#5D4505` | Warning text |

Accent reserved for: primary completion CTA, selected decision/evidence row, active task highlight, focus/pressed state where supported, and success confirmation text. Accent is never reserved for all interactive elements.

State must not depend on color alone. Pair every warning, critical, success, disabled, and blocked state with explicit text such as `Atrasada`, `Presenca precisa ser conferida`, `Rebaixa reprovada`, or `A tarefa continua aberta`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Solicitar rebaixa` |
| Early exception CTA | `Solicitar rebaixa antecipada` |
| Presence gate CTA | `Conferir presenca antes da rebaixa` |
| Approval CTA | `Aprovar rebaixa` |
| Rejection CTA | `Reprovar rebaixa` |
| Application CTA | `Registrar etiqueta aplicada` |
| Final confirmation CTA | `Confirmar etiqueta na area de venda` |
| Empty state heading | `Nenhuma rebaixa pendente` |
| Empty state body | `Quando um lote entrar na janela de rebaixa, a proxima etapa aparecera em Hoje.` |
| Error state | `Nao foi possivel avancar a rebaixa. Confira a presenca fisica do lote e tente novamente.` |
| Delayed approval | `Aprovacao de rebaixa atrasada` |
| Delayed application | `Aplicacao de rebaixa atrasada` |
| Delayed final confirmation | `Conferencia da etiqueta atrasada` |
| Destructive confirmation | `Reprovar rebaixa: informe o motivo. Esta rebaixa sera encerrada e o lote voltara ao monitoramento.` |

Decision and evidence copy:

| Context | Copy |
|---------|------|
| Rule-window request reason | `Janela de rebaixa` |
| Early exception reason label | `Por que esta rebaixa esta sendo pedida antes da janela?` |
| Required rejection reason | `Motivo da reprovacao` |
| Application evidence title | `Comprove a etiqueta aplicada` |
| Application photo option | `Registrar foto da etiqueta/preco` |
| Final evidence title | `Comprove na area de venda` |
| Final photo option | `Registrar foto da etiqueta na area de venda` |
| No photo group | `Sem foto` |
| No photo custom field | `Descreva o motivo sem foto` |
| Push acknowledgement reminder | `Recebimento confirmado. A tarefa continua aberta ate a resolucao fisica.` |

Copy rules:

- Use concrete verb + noun labels, never `OK`, `Submit`, `Salvar`, or `Cancelar` as the primary decision.
- Avoid blame. Describe the missing operational fact and the next action.
- Use `Hoje` as the source-of-truth language: reminders and alerts can cobrar, but completion happens in the task panel.
- Rejection must sound terminal and deliberate, not like a temporary skip.

## Accessibility, Loading, And Edge States

| State | Required UI behavior |
|-------|----------------------|
| Loading active tasks | Keep existing content stable; disable the current action label with progress copy such as `Atualizando tarefas`. |
| Empty markdown queue | Show the empty state copy above and keep lot registration/recent-lot paths reachable from Hoje. |
| Presence uncertain | Show `StatusNotice` before markdown actions and route to observation. |
| Validation error | Put the problem near the field or decision row and expose it through alert text. |
| No-photo path | Require either a predefined no-photo reason or non-empty custom reason before the stage completes. |
| Rejection | Require rejection reason before enabling `Reprovar rebaixa`. |
| Delayed escalated stage | Show stage label, responsible label, escalation audience, and that the task is still open. |
| Stale push-open task | Use existing stale task copy and send the user back to current Hoje tasks. |

Motion: use only press feedback and state transitions that complete in 150-200ms. No page-load choreography. Reduced motion must remain equivalent because content is visible without animation.
