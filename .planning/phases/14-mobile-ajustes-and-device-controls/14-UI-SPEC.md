# Phase 14 - UI Design Contract

> Visual and interaction contract for the mobile Ajustes area. Generated by gsd-ui-phase and verified before plan-phase.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | React Native manual design system |
| Preset | not applicable |
| Component library | Existing mobile `capture-ui.tsx` primitives: `ScreenHeader`, `StatusNotice`, `PrimaryAction`, `SecondaryAction`, `DestructiveAction`, `Field`, `SelectionRow` |
| Icon library | none for Phase 14; do not introduce icon-only controls |
| Font | React Native system sans |

**Source of truth:** new Ajustes UI must reuse `capture-theme.ts`, `capture-ui.tsx`, `mobile-status.ts`, `offline-sync-ui.tsx`, `today-copy.ts`, and the authenticated shell in `AuthGate.tsx`. Do not introduce shadcn, third-party registries, decorative charts, debug-table styling, glass effects, gradient text, or marketing-style cards.

**Product scene:** a store operator checks this screen one-handed in a bright aisle to answer "este aparelho esta pronto para operar?" before returning to `Hoje`, a task, a lot detail, or shift close.

## Screen Contract

### Entry And Return

| Area | Contract |
|------|----------|
| Entry point | Add a fixed `Ajustes` action in the authenticated session bar beside store, role, and session identity. It replaces scattered `Privacidade` and `Sair` shortcuts as the control home. |
| Route behavior | Opening Ajustes preserves the current `CaptureApp` route stack. Back/return restores the exact previous route: `Hoje`, task resolution, lot detail, shift close, product flow, or recent lots. |
| First screen focal point | The visual anchor is the device readiness summary: account/store identity first, then readiness cards for push/lembretes, sync, update/build, privacy/account, and sign-out. |
| Density | Use a single vertical scroll. Avoid nested cards. Each readiness card has one title, one status line, one body/metadata group, and at most two visible actions. |

### Primary Screen Sections

| Section | State Shown | Primary Action | Secondary/Inline Action |
|---------|-------------|----------------|-------------------------|
| Conta e loja | Person, active store, active role, account status, session state | none, read-only | "Falar com lideranca" guidance when role/store is wrong |
| Push e lembretes | Permission, provider/local reminder state, last safe test outcome when available | "Ativar alertas do turno" or "Enviar teste neste aparelho" | "Desativar neste aparelho" when active |
| Sincronizacao | Last central read, last sync/send attempt, pending command count, conflict count, safe-close blocker statement | "Sincronizar pendencias" | "Revisar conflito" or destructive discard inside conflict panel |
| Atualizacao do app | Installed version/build, approved artifact, compatibility, environment, API target, package id | "Ver passo de atualizacao" | none unless planning finds an existing safe deep link |
| Privacidade | Data categories used for operation, evidence, sync, permissions, device/build, audit | "Abrir Centro de Privacidade" | none |
| Sair | Session end, pending work warning | "Sair da conta" after confirmation | "Continuar nos Ajustes" |

### Readiness Vocabulary

| Verdict | Tone | Meaning In Ajustes |
|---------|------|--------------------|
| Apto | success | This device has no active blocker for normal operation. It still does not prove physical execution by itself. |
| Atencao | warning | Operation can continue, but a degraded state needs a next action or blocks a specific validation proof. |
| Bloqueado | critical | This device state blocks safe close, validation, or normal mobile operation until resolved. |

Use existing `mobile-status.ts` descriptors where possible. Never use "seguro" copy for transport-only sync, local cache, or push receipt. Safe-area language is allowed only when central read, no critical blockers, and physical checklist conditions are satisfied by the owning workflow.

## Spacing Scale

Declared values for new Ajustes layout (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline label/detail gaps and badge text gaps |
| sm | 8px | Compact group spacing, readiness card internal rows |
| md | 16px | Default screen padding, card padding, section gaps |
| lg | 24px | Header-to-content spacing and major section separation |
| xl | 32px | Long-form privacy/account group separation |
| 2xl | 48px | Minimum touch target height and major action blocks |
| 3xl | 64px | Reserved for page-level empty space only; avoid in ordinary cards |

Exceptions: none for new layout spacing. Existing reused primitives may keep their encapsulated styles, but new Ajustes styles must not introduce additional spacing values.

## Typography

Use only these type sizes and these two weights in new Ajustes UI:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label / metadata | 14px | 400 or 600 | 20px |
| Body | 16px | 400 | 24px |
| Section heading / card title | 20px | 600 | 25px |
| Screen title | 28px | 600 | 34px |

Rules:
- Body and metadata use 400 unless emphasis is needed for status labels.
- Buttons and state labels use 600.
- No display font, no fluid type, no negative letter spacing.
- Long operational copy wraps under 75 characters per line when possible.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F7F9F6` | Screen background and calm operational canvas |
| Secondary (30%) | `#FFFFFF`, `#EDF3EE` | Readiness cards, account panels, neutral grouped rows |
| Accent (10%) | `#166534` | Primary CTA, active/focused session action, Apto/success state, approved-current build badge |
| Destructive | `#A1271D` | Sign-out confirmation, discard offline action, Bloqueado/critical copy and borders |

Accent reserved for: primary CTA, active/focused navigation/action state, success/Apto status, approved-current build compatibility, and focus indicators. Accent must not be applied to every tappable row.

Semantic supporting colors:
- Warning/Atencao: use `#FFF7E2`, `#E7D39A`, `#5D4505`.
- Critical/Bloqueado: use `#FFF0EE`, `#E8B5B0`, `#A1271D`.
- Neutral borders/text: use `#BAC8BE`, `#3E5547`, `#132018`.

Color cannot be the only signal. Every status includes a text label and consequence sentence.

## Interaction States

### Push And Reminders

| State | UI Treatment | Action |
|-------|--------------|--------|
| `not_requested` | Atencao card with education copy | `Ativar alertas do turno` |
| `active` | Apto card; show permission active and local/remote availability | `Enviar teste neste aparelho`; allow `Desativar neste aparelho` |
| `local_only` | Atencao card; explain local reminder only | `Enviar teste neste aparelho` if local schedule is available |
| `denied` | Atencao card; tasks remain in Hoje | `Abrir configuracoes do aparelho` when supported |
| `unavailable` or `failed` | Atencao by default; Bloqueado only when proving remote push | `Tentar novamente` |

Safe push test result copy must say it proves only this device path. It never says provider global, task execution, or safe area were proven.

### Sync And Conflict Recovery

| State | UI Treatment | Action |
|-------|--------------|--------|
| Empty queue | Apto/neutral summary; no large success celebration | none |
| Pending non-critical commands | Atencao summary; visible count and oldest item | `Sincronizar pendencias` |
| Critical pending command | Bloqueado for safe close; show consequence | `Sincronizar pendencias` |
| Critical conflict | Bloqueado; conflict appears before pending rows | `Revisar conflito` |
| Failed retry | Atencao or Bloqueado based on urgency | `Tentar sincronizar novamente` |
| Discard allowed | Destructive panel with reason field | `Descartar acao offline` disabled until reason is non-empty |

`Ultima leitura central` and `Ultima sincronizacao enviada` must appear as separate rows. If either is missing, show `Sem leitura central confirmada` or `Sem envio confirmado` instead of hiding the field.

### Build And Update Truth

| Compatibility | UI Treatment | Next Step |
|---------------|--------------|-----------|
| `atual` | Apto/success badge using accent soft surface | Continue operating; no update action required |
| `desatualizado` | Atencao badge | Install approved APK manually before validation |
| `desconhecido` | Atencao badge | Reopen app or check installed APK before validating |
| `incompativel` | Bloqueado badge | Do not use for validation; install approved APK |

Display exact public-safe fields: installed version, installed build, approved version, approved build, approved artifact label, environment, API target, package id. Masking already handled by `readMobileBuildInfo`; do not add raw build URLs or tokens.

### Account, Privacy, And Sign-Out

| State | UI Treatment | Action |
|-------|--------------|--------|
| Active session | Read-only identity panel with person/store/role/status | none |
| Wrong store/role suspected | Atencao copy, no manual switcher | guidance to leadership/admin |
| Privacy | Short operational summary plus shortcut | `Abrir Centro de Privacidade` |
| Sign-out with no pending work | Destructive confirmation still required | `Sair da conta` |
| Sign-out with pending sync/conflict | Warning before destructive action | `Sair da conta` only after confirmation; pending work remains pending |

## Accessibility And Responsive Contract

- All tappable controls in Ajustes use at least 48px height.
- The session bar `Ajustes` action must have accessibility label `Abrir Ajustes do aparelho`.
- Status cards use text labels `Apto`, `Atencao`, or `Bloqueado`; color is secondary.
- Critical and warning `StatusNotice` content keeps `accessibilityRole="alert"` behavior from existing primitives.
- Destructive panels require visible body copy before the destructive button.
- Android back from Ajustes returns to the previous operational route. Android back inside a conflict review closes the conflict panel first.
- Long labels wrap; no horizontal scrolling or clipped text in 360px-wide mobile viewport.
- Reduced motion: no decorative page-load animation. State updates may use instant replacement or simple 150-250ms opacity transitions only if existing React Native patterns support it.

## Implementation Boundaries For Planner

The planner should keep these source boundaries explicit:

- `AuthGate.tsx` owns the authenticated session bar and the Ajustes entry action.
- `CaptureApp.tsx` owns route preservation and the Ajustes screen route or shell handoff.
- `capture-ui.tsx`, `capture-theme.ts`, and `mobile-status.ts` own visual primitives, colors, and readiness language.
- `offline-sync-ui.tsx` owns conflict review and destructive discard patterns.
- `build-info.ts` owns installed versus approved build truth and public-safe labels.
- `alert-channel.ts`, `repository.ts`, and existing tests own push state and sync data sources.

Do not plan a manual store switcher, automatic APK updater, provider-global proof, or a new daily diagnostics dashboard in Phase 14.
