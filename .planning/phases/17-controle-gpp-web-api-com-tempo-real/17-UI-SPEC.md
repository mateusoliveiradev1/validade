# Phase 17 - UI Design Contract

> Visual and interaction contract for the web Controle GPP surface, generated by gsd-ui-phase and verified before plan-phase.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn v4 configured in `apps/web/components.json` |
| Preset | `radix-nova`, neutral base, preset code `b2fA` |
| Component library | Existing shadcn/Radix primitives: badge, button, input, select, sheet, skeleton, table, alert-dialog |
| Icon library | lucide-react |
| Font | Geist Variable via `@fontsource-variable/geist` and `--font-sans` |

**Source of truth:** `apps/web/src/index.css`, `apps/web/src/shell/AppShell.tsx`, `apps/web/src/command-center/*`, and existing `apps/web/src/components/ui/*` primitives. Do not introduce a new palette, display font, card style, motion system, or dashboard framework.

**Product scene:** GPP uses this web surface at the store desk or back-office station to clear physical labels/caixas by sector while the sector team continues working in the aisle. The screen must make the next baixa/divergencia action obvious, keep central confirmation explicit, and avoid any decorative dashboard noise.

## Screen Contract

### Route And Shell

| Area | Contract |
|------|----------|
| Route label | `Controle GPP` |
| Route availability | Show only when `controle_gpp_enabled` is true and the session has GPP read capability for the active store. |
| Shell placement | Add as a first-class operational route in `AppShell`, near `Operacao`; do not hide it under `Auditoria` or `Acessos da loja`. |
| First route for `gpp` role | If the active role is `gpp`, default the web app to `Controle GPP` instead of `Operacao`. |
| Non-GPP roles | `lead` and `admin` may see the route only when capability allows. Collaborator web access remains capability-gated. |
| Disabled/access state | If feature flag or capability blocks the route, the nav item is hidden or disabled with an explicit reason. Never show an empty operational shell that looks broken. |

### Page Header

| Element | Contract |
|---------|----------|
| Title | `Controle GPP - {storeName}` |
| Kicker | `Fila da loja` or `Controle GPP` in `text-primary`, one line only. |
| Body copy | One sentence: this is the central queue for avarias, compras internas, divergencias, and history. |
| Status strip | Show central freshness and realtime state together: `Atualizado ha Xs`, `Tempo real ativo`, `Tempo real pausado`, or `Central indisponivel`. |
| Search | Global search by product name, code, setor, or entry id. Search is secondary to tabs and sector grouping. |
| Refresh CTA | `Atualizar` with `RefreshCw`; disabled/loading text is `Atualizando...`. |

Header layout: desktop uses a flex row with title/status on the left and search/refresh on the right. Mobile web stacks into title, status, then search and refresh. Do not use a hero, statistics banner, gradient, or decorative metric cards.

### Tabs

Use a familiar tab/segmented control pattern with these exact tabs:

| Tab | Purpose | Default Empty State |
|-----|---------|---------------------|
| `Avarias` | Main GPP baixa queue by setor and grouped product/code. | `Nenhuma avaria pendente agora` |
| `Compras internas` | Good-product requests from sectors for GPP to attend. | `Nenhuma compra interna pendente agora` |
| `Divergencias` | Items marked by GPP as needing correction/review. | `Nenhuma divergencia aberta agora` |
| `Historico` | Audit/history and completed/canceled/estornado records. | `Nenhum registro no periodo selecionado` |

The tab row stays below the header. On mobile web, tabs may horizontally scroll only if labels remain fully visible and focusable; prefer wrapping into two columns before clipping text.

## Avarias Tab Contract

### Primary Layout

| Layer | Contract |
|-------|----------|
| Sector list | Sectors are the first-level structure. The sector with the highest pending workload opens automatically on first load. |
| Sector summary | Each sector header shows pending count, total quantity/unit summary, divergences, and baixadas today. |
| Group rows | Inside a sector, rows are grouped by `codigo/produto`, not by timestamp. |
| Density | Use bordered rows inside a single sector panel. Avoid card-inside-card nesting. |
| Sorting | Pending first, then divergences indicator, then latest activity, then product/code alphabetic fallback. |

Sector panel style:
- `rounded-lg border border-border bg-card`
- header padding `16px`
- row padding `12px 16px`
- row divider `border-t border-border`
- no shadow decoration.

### Group Row Content

Every avaria group row must show, without opening details:

| Field | UI Treatment |
|-------|--------------|
| Product code | Strong text, first token in the row, e.g. `162`. |
| Product name | Strong text beside code; wraps cleanly. |
| Sector | Secondary metadata unless already in the sector header. |
| Finality | Badge or compact text: `Baixa GPP`, `Reaproveitamento`, `Producao interna`, `Transferencia`. |
| Total quantity | Strong numeric text with unit, e.g. `3,250 kg`; never color-only. |
| Entry count | Compact metadata: `{N} lancamentos`. |
| Divergence count | Warning/critical badge only when greater than zero. |
| Latest activity | `Hoje HH:mm` or date label. |
| Actions | `Baixar`, `Detalhes`, `Divergencia`. |

Action hierarchy:
- `Baixar` uses primary button only when the group is eligible and not divergent.
- `Detalhes` uses outline button.
- `Divergencia` uses warning/critical tone but not destructive red unless it blocks baixa.
- If not eligible, replace `Baixar` with disabled button and visible reason text.

### Group Baixa Confirmation

Group baixa uses `AlertDialog` or an equivalent confirmation surface, not an inline one-click action.

Confirmation must show:
- setor;
- code/product;
- total quantity/unit;
- finality;
- number of entries;
- remaining saldo when relevant;
- warning: `Depois de baixar, ajustes exigem estorno ou correcao administrativa.`

Primary action copy: `Confirmar baixa GPP`.
Cancel action copy: `Voltar para conferir`.
While saving: `Salvando na central...`.
Success feedback: `Baixa registrada na central`.
Failure feedback: `Falha na central. Tente novamente antes de considerar baixado.`

### Details Side Panel

`Detalhes` opens a `Sheet` side panel on desktop and a full-width sheet on mobile web.

Panel structure:
1. Group summary: code/product, setor, total, finality, status, saldo.
2. Individual entries: creator, quantity/unit, reason, movement/finality, created time, central status.
3. Linked movements: reaproveitamento, producao interna, transferencia, and saldo calculation.
4. Audit/history: created, edited, divergence, correction, review, baixa, estorno.
5. Fixed footer actions: `Baixar`, `Divergencia`, `Fechar`.

Focus returns to the row that opened the sheet. The sheet must not contain another card shell around every section; use section headings, dividers, and compact rows.

### Divergence Flow

`Divergencia` opens a side panel or alert-dialog-style form that keeps the user in the queue context.

Required fields:
- closed reason: `Quantidade diferente`, `Codigo/produto errado`, `Etiqueta fisica nao encontrada`, `Setor destino errado`, `Duplicado`, `Producao sem finalidade clara`, `Outro`;
- observation, required;
- optional corrected quantity/product/sector fields when reason requires them.

Action copy:
- primary: `Marcar divergencia`;
- cancel: `Voltar para fila`.

After success, the item leaves the normal eligible baixa queue and appears in `Divergencias`. The visible feedback must say `Divergencia registrada na central`.

## Compras Internas Tab Contract

### Layout

Compras internas use the same sector-first rhythm as `Avarias`, but rows represent requests instead of avaria groups.

Each request row shows:
- setor solicitante;
- product name/description;
- product code if supplied;
- quantity/unit;
- finality, e.g. pizza, preparo, salada;
- requester;
- request time;
- status.

Actions:
- `Atendido`;
- `Parcial`;
- `Sem produto`;
- `Detalhes`.

### Attend Flow

The GPP attend form must require central confirmation.

| Action | Required fields | Success copy |
|--------|-----------------|--------------|
| `Atendido` | confirmed product/code if missing, attended quantity/unit | `Compra interna atendida na central` |
| `Parcial` | attended quantity/unit, reason | `Atendimento parcial registrado na central` |
| `Sem produto` | short reason | `Sem produto registrado na central` |
| `Cancelado` | reason, only where permission allows | `Solicitacao cancelada na central` |

If product code is missing, the form leads with `Confirmar codigo do produto` but does not block viewing the request. It blocks final attendance until GPP confirms or links the product/code.

## Divergencias Tab Contract

The `Divergencias` tab is a work queue, not just a filter.

Rows show:
- current problem reason;
- original group/request;
- who marked divergence;
- age of divergence;
- owner next action: `Setor precisa corrigir`, `Lideranca precisa revisar`, or `GPP pode revisar`;
- correction status.

Allowed actions by capability:
- `Revisar correcao`;
- `Manter divergencia`;
- `Baixar apos revisao`;
- `Abrir detalhes`.

Do not allow `Baixar` while status is still divergent. The primary action becomes available only after correction is marked and GPP review confirms it.

---

## Historico Tab Contract

Historico is a dense investigation list using existing audit/table patterns.

Required filters:
- date range;
- setor;
- product/code;
- type/status;
- actor when available.

Rows show:
- event/action;
- product/code or request label;
- setor;
- actor role;
- occurred/central timestamp;
- status.

Details reuse the same side panel vocabulary from avaria details. Historico must not open a separate dashboard with charts or decorative metrics.

## Central Feedback And Realtime Contract

### Write Feedback

| State | UI Copy | Treatment |
|-------|---------|-----------|
| Idle | action label, e.g. `Baixar` | Enabled only when permission and status allow. |
| Saving | `Salvando na central...` | Disable the initiating action and keep cancel/back disabled only if leaving would create ambiguity. |
| Success | `Registrado na central` or action-specific success | Show inline status/alert and refresh central snapshot. |
| Central failure | `Falha na central` | Keep item visible and retryable; do not mark as success. |
| Permission failure | `Acao indisponivel para este papel` | Explain role/capability without exposing backend internals. |

No optimistic success. No toast-only success for critical writes. Every write result needs visible inline feedback in the panel or row context.

### Realtime

| State | UI Contract |
|-------|-------------|
| Connected | Show compact `Tempo real ativo` status near freshness text. |
| Event received | Do not mutate truth from event payload. Show subtle refresh state and re-read central snapshot. |
| Paused/disconnected | Show `Tempo real pausado` and keep `Atualizar` visible. |
| Publish failed after commit | User still sees central success if API commit succeeded; other clients recover by refresh/polling. |
| Central unavailable | Show `Central indisponivel`; block baixa/attendance actions until central read/write is available. |

Refresh timestamp uses `aria-live="polite"`. Realtime events must not produce urgent screen-reader chatter for every passive update.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge internal gaps, metadata separators |
| sm | 8px | Row sub-groups, status strip gaps, compact form fields |
| md | 16px | Panel padding, row horizontal padding, default form gaps |
| lg | 24px | Header-to-tabs, tab-to-content, major queue sections |
| xl | 32px | Desktop page-level grouping and side-panel section breaks |
| 2xl | 48px | Empty state vertical breathing room and fixed footer offset |
| 3xl | 64px | Reserved for rare page-level separation; avoid inside queue panels |

Exceptions: existing button dimensions (`h-10`, `h-11`, `size-11`, `min-h-12`) remain component dimensions, not spacing tokens.

## Typography

Use fixed product UI sizes only:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Metadata | 14px | 400 | 20px |
| Label | 14px | 600 | 20px |
| Body | 16px | 400 | 24px |
| Row primary | 16px | 600 | 22px |
| Section heading | 20px | 600 | 24px |
| Screen title | 28px | 600 | 34px |

Rules:
- Geist Variable only.
- Weights 400 and 600 only.
- No fluid type, display font, negative tracking, or uppercase-wide eyebrow system.
- Long operational body copy stays under 75ch where possible.
- Numeric quantities should align visually in row layouts; use consistent unit formatting.

## Color

Use existing OKLCH tokens from `apps/web/src/index.css`.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` `oklch(0.972 0.011 145)` | Route canvas and quiet whitespace around work queues |
| Secondary (30%) | `--card`, `--sidebar`, `--muted` | Queue panels, sector headers, side panels, filters |
| Accent (10%) | `--primary` `oklch(0.415 0.126 149)` | Primary action, active tab/route, focus ring, connected/success state |
| Warning | `--warning`, `--warning-border`, `--warning-foreground` | Divergence attention, missing code, partial attendance, realtime paused |
| Destructive/Critical | `--destructive`, `--critical-surface`, `--critical-border` | Central failure, blocked baixa, unauthorized critical action, irreversible administrative correction |

Accent reserved for: active `Controle GPP` route, active tab indicator, `Baixar` when eligible, `Confirmar baixa GPP`, `Atendido`, focus-visible ring, and `Tempo real ativo`. Do not use accent for every badge, every table count, or decoration.

State cannot rely on color alone. Every warning/critical/success state includes a label and a consequence sentence.

## Responsive And Accessibility Contract

- Desktop keeps fixed side nav and max content width `1440px`.
- Desktop `Avarias` uses sector panels in one main column; details open in side sheet instead of navigating away.
- Mobile web stacks header, search, tabs, sector panels, and actions in a single column.
- Action groups wrap before overflowing; buttons keep at least `h-10`, with critical actions using `min-h-12` where confirmation is required.
- No horizontal scrolling for queue rows. If data is too wide, move metadata to second line.
- Active route uses `aria-current="page"`.
- Tabs use `role="tablist"`/`role="tab"` or equivalent accessible shadcn/Radix semantics.
- Side sheets trap focus, close with Esc, return focus to the opener, and keep a labelled heading.
- Confirmation dialogs include `aria-describedby` for the consequence/warning copy.
- Central failure regions use `role="alert"`.
- Freshness and realtime status use `aria-live="polite"`, not assertive.
- Status is never color-only; every state includes text and consequence.
- Respect `prefers-reduced-motion`; use only 150-250ms state transitions where existing components already do.
