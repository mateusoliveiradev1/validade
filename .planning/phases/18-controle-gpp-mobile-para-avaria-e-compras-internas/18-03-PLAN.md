<objective>
Implement the mobile Registrar avaria guided flow.

Purpose: Sector operators must register avarias quickly while GPP receives complete, contract-valid central records. This plan adds the product-code-first wizard, required quantity/unit/finality/destination validation, review step, and central/offline feedback for avaria submissions.
Output: From the Controle GPP hub, `Registrar avaria` opens a short mobile flow that blocks incomplete avaria records, submits valid records to central, shows success only after central acknowledgement, and saves local pending only for real offline/transport failure.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator-entered avaria data to central contract | Required fields must be complete before central submission. |
| Avaria form to local pending queue | Only transport failure may save local pending state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-07 | Tampering | Avaria form validation | mitigate | Block missing product code, quantity/unit, finality, and destination before request creation. |
| T-18-08 | Repudiation | Avaria idempotency | mitigate | Generate and preserve idempotency key for central write and local pending retry. |
| T-18-09 | Spoofing | Avaria central feedback | mitigate | Render success only from parsed `central_confirmed` or `replayed` responses. |
</threat_model>
