<objective>
Implement the mobile Solicitar compra interna flow and the first `Minhas pendencias` status surface.

Purpose: Sector operators often know the needed product by name/description before GPP confirms a code. This plan adds the purchase request flow with optional code, central-confirmed feedback, and a simple pending/status list without turning mobile into the full GPP attendance desk.
Output: From the Controle GPP hub, `Solicitar compra interna` creates central-backed purchase requests, and `Minhas pendencias` shows sector-facing request states plus local pending/conflict states.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Purchase draft to central GPP API | Product code may be missing, but description/quantity/finality must be complete. |
| Central purchase statuses to mobile sector UI | Mobile presents simple read-only statuses without becoming the GPP attendance surface. |
| Local pending to sent-today list | Local-only records must not appear as centrally sent records. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-10 | Tampering | Purchase form validation | mitigate | Block missing description, quantity/unit, and finality while keeping product code optional. |
| T-18-11 | Spoofing | Sent-today projection | mitigate | Include only central-confirmed/replayed records in `Enviadas hoje`. |
| T-18-12 | Elevation of privilege | Mobile pending statuses | mitigate | Keep mobile pending/status view read-focused; do not add detailed baixa/divergence/full audit attendance actions. |
</threat_model>
