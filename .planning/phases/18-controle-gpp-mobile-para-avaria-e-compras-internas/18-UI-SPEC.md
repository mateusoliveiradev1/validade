# Phase 18 - UI Design Contract

> Visual and interaction contract for the mobile Controle GPP surface. Generated by gsd-ui-phase and verified before plan-phase.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none for mobile; use existing React Native primitives |
| Preset | not applicable |
| Component library | Existing `apps/mobile/src/capture/capture-ui.tsx` primitives: `ScreenHeader`, `PrimaryAction`, `SecondaryAction`, `DestructiveAction`, `Field`, `SelectionRow`, `StatusNotice` |
| Icon library | none required for Phase 18 mobile; if navigation needs an icon, use the existing mobile shell pattern and accessible text label |
| Font | React Native system sans stack through existing `StyleSheet` usage |

**Source of truth:** `apps/mobile/src/capture/capture-theme.ts`, `apps/mobile/src/capture/capture-ui.tsx`, `apps/mobile/src/capture/TodayScreen.tsx`, `apps/mobile/src/capture/offline-sync-ui.tsx`, and `apps/mobile/src/capture/today-copy.ts`.

**Product scene:** a sector operator or GPP user is in the aisle with one hand free, under bright store lighting and unstable connectivity. The UI must make the next GPP action obvious, keep central confirmation honest, and never imply that GPP received a record when it is only saved on the device.

## Mobile Surface Contract

### Route And Entry

| Area | Contract |
|------|----------|
| Route label | `Controle GPP` |
| Hub title | `Controle GPP` |
| Route availability | Show only when `controle_gpp_enabled` is true and the session has `canCreateGppEntry` or `canReadGppQueue`. Backend authorization remains mandatory. |
| First route for `gpp` role | A session with active role `gpp` opens directly into `Controle GPP` when the feature flag and capability allow it. |
| Other roles | Collaborator, lead, and admin keep `Hoje` as the first daily surface and open Controle GPP through a separate app entry. |
| Today boundary | Do not place `Registrar avaria` or `Solicitar compra interna` inside `TodayScreen`. Phase 19 owns Hoje integration. |
| Disabled/access state | If hidden by feature flag or capability, no empty GPP shell is shown. If an API write returns permission/flag denial, show central failure copy and do not enqueue local success. |

### Hub Layout

The first screen is a one-column `ScrollView` with `captureColors.background`, `gap: 16px`, `padding: 16px`, and `paddingBottom: 32px`.

| Zone | Contract |
|------|----------|
| Header | `ScreenHeader` title `Controle GPP`; body explains that avarias and compras internas are sent to central GPP. |
| Primary action block | Two full-width actions: `Registrar avaria` and `Solicitar compra interna`. Both have min height 48px. |
| Follow-up block | `Minhas pendencias` row first when there are local pending records, conflicts, or central purchases requiring user review. |
| Sent-today block | `Enviadas hoje` summary row with count and last central acknowledgement time when available. |
| Sync status | Inline `StatusNotice` or queue summary below the hub actions when any record is `Pendente neste aparelho`, retrying, failed, or in conflict. |

No hero, metric banner, decorative chart, nested cards, gradient, or dashboard-style stats. Counts are compact metadata, not the focal point.

## Visual Hierarchy

| Screen | Focal Point | Hierarchy Contract |
|--------|-------------|--------------------|
| Hub | The next physical action: `Registrar avaria` or `Solicitar compra interna` | Title, short body, primary action block, then pending/sent summaries. Pending/conflict notice moves above sent summaries when present. |
| Registrar avaria | Current required step in the guided flow | Product code first, then confirmation name, then quantity/unit, finality/destination, review, central submission. |
| Solicitar compra interna | Product description/name and quantity request | Description first, optional code secondary, then quantity/unit, finality, review, central submission. |
| Minhas pendencias | Records that still require attention | Conflicts first, then local pending, then central statuses. Every row shows item, quantity/unit, state, and time. |
| Enviadas hoje | Confirmed central feedback | Show only central-confirmed or replayed records as sent. Local pending records stay out of this success list. |

Visual rules:
- Primary actions use `captureColors.accent` only for the one next action in the current step.
- Selection rows use `captureColors.surfaceMuted` by default and `captureColors.accentSoft` only for the selected option.
- Warnings and conflicts use existing warning/critical surfaces with text, not color alone.
- Long product names wrap within the row; never require horizontal scrolling.

## Registrar Avaria Flow

Use a short guided flow. The executor may implement separate route states or an internal step state, but the visible order is locked:

1. Product code.
2. Product name/description confirmation.
3. Quantity and unit.
4. Finality and destination.
5. Review and submit to central.

| Step | Contract |
|------|----------|
| Product code | Required. Label `Codigo do produto`. Product name is supporting confirmation copy, not the primary key. |
| Quantity | Required positive quantity plus one unit from `un`, `kg`, `g`, `l`, `ml`, `caixa`, `pacote`. Free-text quantity is not accepted. |
| Finality | Required selection with exact labels: `Baixa GPP`, `Reaproveitamento`, `Producao interna`, `Transferencia`. |
| Destination | Required when the selected finality needs a destination or operational finality detail. |
| Review | Shows code, name, quantity/unit, finality, destination, and the central submission consequence. |

Primary CTA by step:
- Product step: `Continuar avaria`.
- Quantity/finality steps: `Continuar avaria`.
- Review step: `Enviar avaria para central`.

Blocking validation copy:
- Missing product code: `Informe o codigo do produto para enviar a avaria ao GPP.`
- Missing quantity/unit: `Informe quantidade e unidade antes de continuar.`
- Missing finality/destination: `Escolha a finalidade e o destino antes de enviar.`

## Solicitar Compra Interna Flow

Use a short guided flow with product description first and product code optional.

| Step | Contract |
|------|----------|
| Product description | Required. Label `Produto solicitado`. Placeholder `Nome ou descricao do produto`. |
| Product code | Optional. Label `Codigo do produto (opcional)`. If present, show as supporting metadata. |
| Quantity | Required positive quantity plus one unit from `un`, `kg`, `g`, `l`, `ml`, `caixa`, `pacote`. |
| Finality | Required text/selection. Label `Finalidade da compra interna`. |
| Review | Shows description, optional code, quantity/unit, finality, requester sector, and central submission consequence. |

Primary CTA by step:
- Product step: `Continuar compra interna`.
- Quantity/finality steps: `Continuar compra interna`.
- Review step: `Enviar compra para central`.

Blocking validation copy:
- Missing product description: `Descreva o produto para o GPP localizar ou confirmar o codigo.`
- Missing quantity/unit: `Informe quantidade e unidade antes de continuar.`
- Missing finality: `Informe a finalidade da compra interna antes de enviar.`

## Pendencias And Sent-Today Contract

### Minhas Pendencias

Rows show the simplest sector-facing status vocabulary:

| Status | UI Copy | Treatment |
|--------|---------|-----------|
| `solicitado` | `Enviada` | Neutral/success only after central acknowledgement. |
| `atendido` | `Atendida` | Success tone with central time. |
| `atendido_parcial` | `Parcial` | Warning tone with reason when available. |
| `sem_produto` | `Sem produto` | Warning tone with reason when available. |
| `cancelado` | `Cancelada` | Neutral or critical only if the user must act. |
| local pending | `Pendente neste aparelho` | Warning tone; must say central has not received it yet. |
| conflict | `Conflito de GPP` | Critical tone; requires review before success. |

Each row includes product code/name or description, quantity/unit, current state, and time. Local pending rows include a `Sincronizar pendencias GPP` action when retry is available.

### Enviadas Hoje

Only records with `GppMutationResponse.state` of `central_confirmed` or `replayed` appear here.

Empty state:
- Heading: `Nenhum envio confirmado hoje`.
- Body: `Avarias e compras aparecem aqui somente depois que a central confirma o recebimento.`

## Central Feedback And Offline Truth

| State | UI Copy | Treatment |
|-------|---------|-----------|
| Idle | action label, e.g. `Enviar avaria para central` | Enabled only when required fields and capability allow. |
| Sending | `Enviando para central...` | Disable the initiating action and keep visible progress in the form context. |
| Central confirmed | `Registrado na central` for avaria; `Solicitacao enviada para central` for purchase | Show inline success and refresh GPP snapshot or affected lists. |
| Replayed | `Registro ja confirmado na central` | Treat as central success without creating a duplicate. |
| Network failure while offline | `Pendente neste aparelho` | Save locally with idempotency key, explain that central has not received it yet, and offer retry/manual sync. |
| Central failure | `Falha na central. Corrija os dados ou tente novamente antes de considerar enviado.` | Keep item in form/review context; do not enqueue as offline success. |
| Permission/feature failure | `Acao indisponivel para este papel.` | Explain role/capability in user-facing terms and do not enqueue local success. |
| Conflict after retry | `Conflito de GPP. Revise antes de reenviar ou descartar.` | Keep visible in `Minhas pendencias`; allow correction/retry or discard with justification. |

Rules:
- Online success appears only after the central API returns `central_confirmed` or `replayed`.
- A transport/network failure is the only path to local `Pendente neste aparelho`.
- Server validation, business-rule rejection, feature-flag denial, and authorization denial are central failures, not offline pending.
- Retry carries the original idempotency key and must not create duplicates.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, helper text gaps, compact metadata spacing |
| sm | 8px | Row sub-groups, button groups, notice internals |
| md | 16px | Screen padding, section gaps, default card/row padding |
| lg | 24px | Header-to-content breaks and flow step groups |
| xl | 32px | Bottom padding and major flow separation |
| 2xl | 48px | Empty-state vertical breathing room and minimum touch target reference |
| 3xl | 64px | Rare page-level separation only; avoid inside forms |

Exceptions: existing component-internal padding may remain inherited for `Field`, `DatePickerAction`, queue rows, and notices. Do not add any new spacing token outside the declared scale in Phase 18 screens.

## Typography

Use fixed product UI sizes only:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label/metadata | 14px | 400 or 600 | 20px |
| Body/action | 16px | 400 or 600 | 24px |
| Section heading | 20px | 600 | 25px |
| Screen title | 28px | 600 | 34px |

Rules:
- Weights 400 and 600 only.
- No fluid type, display font, negative tracking, or uppercase-wide eyebrow system.
- Field labels use 14px/600; helper and validation text use 14px/400.
- Primary values such as product code, quantity, and current state use 16px/600 or 20px/600 depending on row density.
- Body copy stays short enough to scan under aisle pressure.

## Color

Use existing mobile tokens from `apps/mobile/src/capture/capture-theme.ts`.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `captureColors.background` `#F7F9F6` | Screen canvas and quiet whitespace around the task |
| Secondary (30%) | `captureColors.surface` `#FFFFFF`, `captureColors.surfaceMuted` `#EDF3EE` | Cards, rows, form sections, queue panels |
| Accent (10%) | `captureColors.accent` `#166534` | Primary action, selected option, active GPP entry, focus/success state |
| Warning | `captureColors.warningSurface` `#FFF7E2`, `captureColors.warningInk` `#5D4505` | Offline pending, partial request, stale/retry state |
| Destructive | `captureColors.critical` `#A1271D`, `criticalSurface` `#FFF0EE`, `criticalBorder` `#E8B5B0` | Central failure, conflict, discard pending record |

Accent reserved for: active `Controle GPP` entry, primary submit action in the current step, selected finality/unit row, focus ring when implemented, central success, and confirmed sent-today markers. Do not use accent for every row count, every status chip, or decoration.

State cannot rely on color alone. Every warning, critical, pending, and success state includes a text label and consequence copy.

| Component/Pattern | Contract |

## Responsive And Accessibility Contract

- Mobile is primary. Every action target is at least 48px high.
- The main content stays in a single column; action groups may wrap but must never overflow horizontally.
- Product code, product name/description, quantity/unit, finality, state, and time must remain visible without opening an extra modal for basic review.
- Use `accessibilityRole="button"` and `accessibilityState` following existing `capture-ui` patterns.
- Central failure and conflict notices use alert semantics through `StatusNotice`.
- Online progress and local pending changes should use polite live-region behavior when available; do not create urgent screen-reader chatter for every retry attempt.
- State is never color-only. Include visible text for pending, success, warning, critical, disabled, and conflict states.
- Reduced motion is respected by relying on existing React Native state transitions; no decorative page-load animation.
- Offline conflict discard requires a text reason before enabling the destructive action.
